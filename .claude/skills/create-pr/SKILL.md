# Creating Pull Requests

Related skills: branch names → `branch-names`; commit messages → `git-commit`;
which tool to open the PR with → the `github-tool-preference` rule (`.claude/rules/github-tool-preference.md`; default: `gh pr create`).

## Before You Open

- Commits follow `git-commit` conventions; the branch is named per `branch-names`
- The branch is pushed to the remote
- The PR targets the correct base branch (usually `main`)

## PR Titles

Use the [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>(<scope>): <short description>
```

**Types:** `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `style`, `ci`
(same set as commits — `git-commit` is canonical for the type list and rules).

Same rules as a commit subject: lowercase, imperative mood, no trailing period, no
`Co-Authored-By` trailers. Scope is optional — use it when the change is isolated to
a clear area.

PR-title specifics:

- **Single-commit PR:** the title matches or summarizes that commit
- **Multi-commit PR:** write a title that captures the overall change, not just the first commit

**Examples:**

```
feat(agents): add model assignments to all agent frontmatter
fix(auth): handle expired token on refresh
docs: update setup instructions for new env vars
```

## PR Body

Use and follow the template in `template.md`.

- Write as a human engineer would — clear, direct, and professional.
- **No AI attribution of any kind:** no "Generated by", "Co-authored by AI", "🤖",
  or any mention of Claude, Anthropic, or code-generation tools. It must read as if
  the engineer who built it wrote it themselves. (Same rule as `git-commit`.)
- Don't list every file changed — reviewers can read the diff.
- Keep it high-level enough that a human can understand what happened; add detail
  only where the change is complex or important.
