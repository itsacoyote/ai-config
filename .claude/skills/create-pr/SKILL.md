## Use the project's conventions first

This skill travels into other projects. Before applying its own guidance, check whether the host project already defines how PRs should be opened, and prefer that. Look in this order and stop at the first that applies:

1. **A documented PR process** in the project's `CLAUDE.md`, `AGENTS.md`, `CONTRIBUTING.md`, or `.github/CONTRIBUTING.md` â€” e.g. required base branch, labels, reviewers, linked-issue syntax, a checklist, or a title convention. Follow it. If it conflicts with this skill (say it mandates a non-Conventional-Commits title), the project's process wins.
2. **A GitHub PR template.** Check, in order: `.github/PULL_REQUEST_TEMPLATE.md`, `.github/pull_request_template.md`, `PULL_REQUEST_TEMPLATE.md` (repo root), `docs/PULL_REQUEST_TEMPLATE.md`, and any files under `.github/PULL_REQUEST_TEMPLATE/` (a directory means multiple templates â€” pick the one that fits the change, or ask). If one exists, **fill out that template** for the body instead of this skill's `template.md`.

## Before You Open

- Commits follow `git-commit` conventions; the branch is named per `branch-names`
- The branch is pushed to the remote
- The PR targets the correct base branch (usually `main`)

## PR Titles

Use the [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>(<scope>): <short description>
```

**Types:** `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `style`, `ci`
(same set as commits â€” `git-commit` is canonical for the type list and rules).

Same rules as a commit subject: lowercase, imperative mood, no trailing period, no
`Co-Authored-By` trailers. Scope is optional â€” use it when the change is isolated to
a clear area.

PR-title specifics:

- **Single-commit PR:** the title matches or summarizes that commit
- **Multi-commit PR:** write a title that captures the overall change, not just the first commit

**Examples:**

```
feat(agents): add model assignments to all agent frontmatter
fix(auth): handle expired token on refresh
docs: update setup instructions for new env vars
```

## PR Body

If the project has a GitHub PR template (see "Use the project's conventions first"), fill that out. Otherwise, use and follow the template in `template.md`.

- Write as a human engineer would â€” clear, direct, and professional.
- **No AI attribution of any kind:** no "Generated by", "Co-authored by AI", "đź¤–",
  or any mention of Claude, Anthropic, or code-generation tools. It must read as if
  the engineer who built it wrote it themselves. (Same rule as `git-commit`.)
- Don't list every file changed â€” reviewers can read the diff.
- Keep it high-level enough that a human can understand what happened; add detail
