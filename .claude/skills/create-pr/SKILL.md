# Creating Pull Requests

Related skills: branch names → `branch-names`; commit messages → `git-commit`;
which tool to open the PR with → the `github-tool-preference` rule (`.claude/rules/github-tool-preference.md`; default: `gh pr create`).

## Step 1 — Run the preflight (discovery)

Run the script first. It does the deterministic discovery so you don't re-derive it:

```bash
sh .claude/skills/create-pr/scripts/pr-preflight.sh                  # report + the body scaffold to fill in
sh .claude/skills/create-pr/scripts/pr-preflight.sh --template-only  # print ONLY the chosen template body
```

It reports three things and prints the body scaffold to fill in:

- **Convention docs present** — which of `CLAUDE.md`, `AGENTS.md`, `CONTRIBUTING.md`,
  `.github/CONTRIBUTING.md`, `docs/CONTRIBUTING.md` exist. **Read the ones it lists** for any
  required PR process (base branch, labels, reviewers, linked-issue syntax, title rules). The
  script can't interpret that process — you must. **If it conflicts with this skill (e.g. a
  non-Conventional-Commits title), the project's process wins.**
- **The PR template**, by GitHub's precedence (`.github/` → repo root → `docs/`, single file
  first, then a `PULL_REQUEST_TEMPLATE/` directory). It prints the chosen template's body as
  the scaffold. If only a **directory** of templates exists, the script lists them and you
  **pick the one that fits the change (or ask)** — that choice is yours, not the script's. If
  no project template exists, it falls back to this skill's `template.md`.
- **Readiness** — current branch, the base/target branch, whether the branch is pushed (with
  the exact `git push` command if not), and whether a PR already exists for the branch (edit
  that one instead of opening a duplicate).

Everything below is the **default** for writing the title and body — a project's own process
and template, surfaced by the script, always take precedence.

## PR Titles

Use the [Conventional Commits](https://www.conventionalcommits.org/) format:

```
<type>(<scope>): <short description>
```

**Types:** `feat`, `fix`, `refactor`, `docs`, `test`, `chore`, `perf`, `style`, `ci`
(same set as commits — `git-commit` is canonical for the type list and rules).

Same rules as a commit subject: lowercase, imperative mood, no trailing period, no
`Co-Authored-By` trailers. Scope is optional — use it when the change is isolated to
a clear area.

PR-title specifics:

- **Single-commit PR:** the title matches or summarizes that commit
- **Multi-commit PR:** write a title that captures the overall change, not just the first commit

**Examples:**

```
feat(agents): add model assignments to all agent frontmatter
fix(auth): handle expired token on refresh
docs: update setup instructions for new env vars
```

## PR Body

Fill in the body scaffold the preflight printed (Step 1) — a project template if one exists, otherwise this skill's `template.md`.

- Write as a human engineer would — clear, direct, and professional.
- **No AI attribution of any kind:** no "Generated by", "Co-authored by AI", "🤖",
  or any mention of Claude, Anthropic, or code-generation tools. It must read as if
  the engineer who built it wrote it themselves. (Same rule as `git-commit`.)
- Don't list every file changed — reviewers can read the diff.
- Keep it high-level enough that a human can understand what happened; add detail
  only where the change is complex or important.
