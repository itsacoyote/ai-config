# Spec: Agent Commit on Handoff

**Date:** 2026-05-21
**Status:** Draft

## Summary

Make every pipeline agent (Define, Research, Plan, Implement, Validate, Document) commit and push its own output files — the step document, any artifacts produced during the step, and the updated `context.yaml` — before returning control to the `/feature` orchestrator. This standardizes how feature state crosses agent boundaries, keeps the remote branch in sync after every step, and removes the silent gap where work generated by one agent is only persisted locally until the next agent (or a future run) happens to commit it.

## Problem Statement

The pipeline currently has inconsistent handoff behavior across agents:

- **Define** pushes the branch and opens a draft PR but does not commit the `1_spec.md` it just wrote — the file sits uncommitted in the working tree until something else picks it up.
- **Research**, **Validate**, and **Document** write their documents and update `context.yaml` but never commit or push.
- **Plan** is the only agent whose post-workflow instructions explicitly tell it to commit (`3_plan.md`), and even Plan does not push.
- **Implement** commits per task during its workflow, but does not push at the end and does not commit `context.yaml` checkpoint updates.

The consequences:

1. **State is lost on interruption.** If a user closes the session between Research and Plan, the `2_research.md` file and any artifacts written to `.docs/<feature>/artifacts/` exist only in the local working tree. Resuming the workflow from another machine, or after a `git stash`/`git reset`, loses everything Research produced.
2. **The remote branch lags reality.** The draft PR opened by Define never gets updated as Research, Plan, Implement, and Validate run. A reviewer looking at the PR mid-workflow sees only the empty initial commit, even though significant work has happened locally.
3. **`context.yaml` desynchronizes from the docs.** Agents update `context.yaml` (artifacts list, checkpoint, completed steps, recommended skills, output artifacts), but those updates are not committed alongside the files they reference. A `git checkout` of an older commit then shows a `context.yaml` that points to artifacts that don't exist at that revision.
4. **Recovery and audit are harder.** There's no per-step commit boundary in git history. If validation finds a problem with the plan, there's no clean revert point — Plan's output is mixed with whatever Implement subsequently produced.

This affects anyone running the multi-agent workflow, on every feature, every time the pipeline crosses an agent boundary.

## Goals

- Every pipeline agent commits the files it produced or modified during its step before returning to the orchestrator.
- Every pipeline agent pushes the feature branch to the remote before returning, so the draft PR reflects current state at every checkpoint.
- `context.yaml` updates are included in the same commit as the document or artifacts they describe, so any commit on the feature branch is internally consistent.
- The post-handoff `context.yaml` update done by the orchestrator (advancing `current_step` / appending to `completed_steps`) is itself committed and pushed, so step transitions are visible in git history.
- Commit messages follow the existing `git-commit` skill convention (Conventional Commits, no AI attribution).

## Non-Goals

- Changing what each agent writes or how it writes it — the file outputs themselves are unchanged.
- Changing the orchestration sequence (Define → Research → Plan → Implement → Validate → Document) or the approval gate.
- Adding new commit hooks or CI checks.
- Squashing or rewriting commits at the end of the pipeline — each step keeps its own commit.
- Implement's per-task commit behavior — that already exists and stays as-is; the new requirement only adds an end-of-step commit for `context.yaml` if Implement modified it after its last task commit, plus a push.
- Handling merge conflicts on push beyond surfacing the failure to the user — automatic conflict resolution is out of scope.
- Changing how the Plan agent recommends skills or how the Validate agent runs reviews.

## User Stories

- As a developer running the pipeline, I want each agent to commit and push its work before handing off, so that if I close my session between steps I can resume from any machine without losing anything.
- As a reviewer of a feature PR, I want to see incremental commits for spec, research, plan, implementation, validation, and documentation, so I can review the feature's evolution and revert any step independently if needed.
- As a developer resuming a feature mid-pipeline, I want `context.yaml` and the docs it references to be consistent at every commit, so checking out an earlier commit gives me a working snapshot of that point in the pipeline.
- As a developer whose Implement step is interrupted partway through, I want the orchestrator's post-handoff `context.yaml` update to be committed and pushed, so the next session can see exactly which step completed last.
- As a developer running the pipeline on a feature whose push fails (e.g. non-fast-forward), I want the agent to stop and surface the failure rather than silently proceeding, so I can resolve the remote state before more work piles up locally.

## Requirements

### Define agent

- After writing `1_spec.md`, commits `1_spec.md` and the updated `context.yaml` together with a conventional commit message (e.g. `docs(spec): add spec for <feature name>`).
- After the commit, pushes the branch with `git push -u origin <feature.branch>` (the existing push step — unchanged).
- Then opens the draft PR (the existing step — unchanged).

### Research agent

- After writing `2_research.md` and saving any artifacts to `<feature.folder>/artifacts/` and updating `context.yaml` (`artifacts` list), commits all of these together with a conventional commit message (e.g. `docs(research): add research for <feature name>`).
- After the commit, pushes the branch with `git push`.

### Plan agent

- After writing `3_plan.md` and updating `recommended_skills` in `context.yaml`, commits both together with a conventional commit message (e.g. `docs(plan): add implementation plan for <feature name>`). This replaces the existing standalone Plan commit.
- After the commit, pushes the branch with `git push`.

### Implement agent

- Keeps its existing per-task commit behavior.
- After the final task commit, if `context.yaml` has uncommitted changes (e.g. the last `workflow.checkpoint` update), commits `context.yaml` with a conventional commit message (e.g. `chore(context): update workflow checkpoint`).
- Pushes the branch with `git push` before returning.

### Validate agent

- After writing `4_validate.md` and updating `context.yaml` (`output_artifacts` list, any further checkpoint), commits both together with a conventional commit message (e.g. `docs(validate): add validation report for <feature name>`).
- After the commit, pushes the branch with `git push`.

### Document agent

- After updating any documentation (README, CLAUDE.md, feature docs, API docs, inline comments, changelog) and updating `context.yaml` (`documentation_created` list), commits all changed files together with a conventional commit message (e.g. `docs: update documentation for <feature name>`).
- After the commit, pushes the branch with `git push`.
- Then updates the PR description and removes draft status (the existing steps — unchanged).

### Orchestrator post-handoff protocol

- The `/feature` orchestrator's post-return protocol (advance `current_step`, append `completed_steps`) is followed by a commit of the modified `context.yaml` and a push, so step transitions are visible on the remote.
- Commit message follows the convention `chore(context): advance workflow to <next step>`.

### Cross-cutting

- All commit messages use the project's existing `git-commit` skill: Conventional Commits, lowercase, imperative mood, no `Co-Authored-By` trailers, no AI attribution.
- Commits are made via the `git-commit` skill invocation pattern already established in CLAUDE.md (skill invoked before `git commit`).
- If `git push` fails (non-fast-forward, network error, auth failure), the agent writes an escalation to `context.yaml` (`workflow.escalated: true`, `workflow.escalation_reason: <push failure detail>`) and returns. The orchestrator halts the pipeline as it does for any other escalation. This matches the existing escalation contract — no new exit path.
- Agents commit only files in `<feature.folder>` and the orchestrator's `context.yaml`. They must not stage or commit unrelated working-tree changes — staging is done by explicit `git add <path>` of the specific files the agent produced, never `git add -A` or `git add .`.

## Constraints

- The project uses Conventional Commits and the `git-commit` skill enforces no AI attribution — every new commit added by this feature must comply.
- The `feature` orchestrator already manages `context.yaml` writes between agents. Any new commit step for orchestrator updates must happen after the orchestrator's existing write, not replace it.
- The `define` agent currently runs `gh pr create --draft`. The PR is created against an empty branch today; after this feature, the PR's first commit will be Define's spec commit, which means Define must commit before pushing for the PR to have meaningful content.
- The Implement agent already commits per task during the implement skill workflow. The new requirements must not introduce duplicate or empty commits — the end-of-step commit is conditional on uncommitted changes to `context.yaml`.
- The `git-commit` skill must be invoked before each `git commit` (per the project's hook enforcement noted in MEMORY.md). Agents must call `Skill(git-commit)` before each commit.

## Acceptance Criteria

- [ ] Define agent's documented workflow includes a commit step for `1_spec.md` + `context.yaml` after the spec is written and before `git push -u origin`.
- [ ] Research agent's documented workflow includes a commit step for `2_research.md`, any artifacts, and `context.yaml` after the research is written, followed by `git push`.
- [ ] Plan agent's documented workflow includes a commit step for `3_plan.md` + `context.yaml` and a `git push` after it. The previous "commit the plan" instruction is updated, not duplicated.
- [ ] Implement agent's documented workflow includes a conditional end-of-step commit for `context.yaml` (only if dirty) and a `git push` before returning.
- [ ] Validate agent's documented workflow includes a commit step for `4_validate.md` + `context.yaml` and a `git push` after it.
- [ ] Document agent's documented workflow includes a commit step for documentation changes + `context.yaml` and a `git push` before the existing PR description update / draft removal.
- [ ] The `/feature` orchestrator's post-return protocol is updated to commit and push `context.yaml` after advancing `current_step` and appending to `completed_steps`.
- [ ] All new commit instructions reference the `git-commit` skill (or are otherwise consistent with the project's hook enforcement).
- [ ] All new commit messages in the instructions follow Conventional Commits and contain no AI attribution.
- [ ] Every agent's instructions explicitly stage only files within `<feature.folder>` and `context.yaml` — never `git add -A` or `git add .`.
- [ ] Push failure handling is specified in every agent's instructions: write to `workflow.escalated` / `workflow.escalation_reason` and return.
- [ ] Running the full pipeline end-to-end on a sample feature produces at least one commit per pipeline step on the feature branch (Define, Research, Plan, Implement, Validate, Document — plus orchestrator transition commits), and the remote `feature/<short-name>` branch matches local `HEAD` after each step.
