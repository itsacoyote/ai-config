# Research: Agent Commit on Handoff

**Spec:** [1_spec.md](1_spec.md)
**Date:** 2026-05-21

## Summary

This feature is entirely a documentation/instruction change inside `.claude/` — no application code, no tests, no schemas. The work surface is six agent instruction files (`.claude/agents/*.md`) and one orchestrator skill (`.claude/skills/feature/SKILL.md`). The standards to comply with (Conventional Commits, no AI attribution, the `git-commit` skill invocation rule) already live in `.claude/skills/git-commit/SKILL.md`, `.claude/skills/create-pr/SKILL.md`, and `CLAUDE.md`, and are enforced at runtime by `.claude/hooks/skill-check.sh`. The research focuses on (1) what each agent's "After the workflow completes" section currently says, (2) where the orchestrator does its post-return `context.yaml` write, (3) which conventions and patterns the new commit/push instructions must match, and (4) the few edge cases the spec leaves ambiguous.

## Gaps: What Needs to Be Created

Nothing brand-new is created — every change is an in-place edit of existing instruction prose. The "gaps" relative to the spec are missing instruction blocks that need to be added to existing files:

- **Define agent — commit step.** Currently the Define agent's "After the workflow completes" section goes: 1. write spec → 2. push branch → 3. open draft PR → 4. return. There is no step that stages and commits `1_spec.md` + `context.yaml` before step 2. The new step must be inserted between today's steps 1 and 2.
- **Research agent — commit + push steps.** The Research agent's "After the workflow completes" section ends after step 2 (artifacts written, `context.yaml` updated). Two new steps must be appended: stage-and-commit `2_research.md`, any artifacts under `<feature.folder>/artifacts/`, and `context.yaml`; then `git push`.
- **Plan agent — broaden commit step and add push.** Today step 3 says "Commit the plan with a conventional commit message" but does not specify which files to stage. It needs to be widened to `3_plan.md` + `context.yaml` and a `git push` step added after it.
- **Implement agent — end-of-step commit (conditional) and push.** Today the agent has no end-of-step instructions beyond the per-task workflow inside `.claude/skills/implement/SKILL.md`. A new section is needed: if `git status` shows `context.yaml` is dirty (last checkpoint update wasn't committed), stage-and-commit it with `chore(context): update workflow checkpoint`; then `git push` unconditionally.
- **Validate agent — commit + push steps.** The Validate agent currently has an "After the workflow completes" section that writes `4_validate.md` and (implicitly) `context.yaml`. Two new steps need to be appended: stage-and-commit those two files; then `git push`.
- **Document agent — explicit commit + push steps before PR ops.** Today the agent's `## PR Description` section says "Once all documentation is updated and committed, update the PR description using `gh pr edit`" — the word "committed" is mentioned but no explicit commit step is written. The instructions need to spell out: stage-and-commit all docs touched (README, CLAUDE.md, feature docs, API docs, inline comments, changelog) and `context.yaml`; `git push`; then proceed to the existing `gh pr edit` and `gh pr ready` steps.
- **Orchestrator post-return protocol — commit + push step.** Today the protocol ends at step 5 "Write the updated `context.yaml`". A step 6 must commit `context.yaml` with `chore(context): advance workflow to <next step>` and a step 7 must `git push`.
- **Push-failure escalation block — added to all six agents and the orchestrator.** Each new push step needs a uniform failure branch: if `git push` exits non-zero, write `workflow.escalated: true` and `workflow.escalation_reason: <push failure detail>` to `context.yaml` and return. This matches the existing escalation contract documented in `.claude/skills/agent-context/SKILL.md` lines 94-109.

## Patterns and Conventions to Follow

Document conventions observed in the codebase that the Planner and implementer should match.

- **Conventional Commits, lowercase, imperative mood, no period, no AI attribution.** Defined in `.claude/skills/git-commit/SKILL.md` and reiterated in `.claude/skills/create-pr/SKILL.md` and `CLAUDE.md`. Every new commit message *string* embedded in agent instructions (whether quoted as an example or as a literal command) must comply. The spec's example messages (`docs(spec): add spec for <feature name>`, `docs(research): add research for <feature name>`, `docs(plan): add implementation plan for <feature name>`, `chore(context): update workflow checkpoint`, `docs(validate): add validation report for <feature name>`, `docs: update documentation for <feature name>`, `chore(context): advance workflow to <next step>`) all comply.
- **Skill invocation before `git commit` and `gh pr create|edit`.** The pre-tool hook at `.claude/hooks/skill-check.sh` injects a `[skill-check]` reminder before any Bash call matching `git commit` or `gh pr create|edit`. The hook does not block — it nags. The convention is therefore: instruction blocks that introduce a new `git commit` must mention "use the `git-commit` skill" (or equivalent), so an agent reading the instructions invokes the skill first and the hook's reminder is satisfied. The Plan agent already follows this implicitly by listing `git-commit` in its frontmatter `skills:` array and saying "Commit the plan with a conventional commit message". The same approach is the cleanest path forward — adding `git-commit` to the `skills:` array of the Research, Validate, and Document agents and using the same phrasing as Plan.
- **Frontmatter `skills:` arrays are the source of truth for which skills an agent can use.** Looking at each agent: Define already has `git-commit`; Plan already has `git-commit`; Implement already has `git-commit`; Research does **not** have `git-commit`; Validate does **not** have `git-commit`; Document does **not** have `git-commit`. The three missing entries must be added — otherwise the agent cannot invoke the skill and the hook reminder will fire without a path to comply.
- **Agent "After the workflow completes" sections use numbered steps with a leading verb.** Define: "Use the spec skill...", "Push the branch...", "Run `gh pr create`...", "Return.". Plan: "Write the plan...", "Scan `.claude/skills/*/SKILL.md`...", "Commit the plan...". Research and Validate use the same numbered-step format. The new steps must blend in — numbered, leading verb, imperative mood, one action per step.
- **Agents stage with explicit `git add <path>`, never `git add -A` or `git add .`.** Acceptance criterion 9 enforces this explicitly. There is no existing instruction in any agent today that uses `git add -A` or `git add .`, so the pattern is already implicit. The new instructions just need to spell out the exact paths each agent stages, so the constraint is documented and not just convention.
- **Escalation via `workflow.escalated` + `workflow.escalation_reason`.** Documented in `.claude/skills/agent-context/SKILL.md` (lines 94-109). Already used by Implement and Validate for code-review and reviewer-loop failures. The push-failure case needs to write the same two fields with the same YAML merge semantics (do not replace other `workflow` fields). The implement and validate agents both show a YAML block example with the `# Merge into existing workflow block — do not replace other fields` comment — reuse that exact comment for consistency.
- **One agent file per task in the plan.** The previous `.docs/2026-05-13-workflow-orchestrator/3_plan.md` modified seven files and structured itself as one task per file plus one task for the template change. The plan for this feature should follow the same pattern: one task per agent file (six tasks), one task for the orchestrator skill, and ideally one final task that runs the full pipeline end-to-end on a throwaway sample feature to satisfy acceptance criterion 11.
- **Numbered steps inside tasks use `- [ ] **Step N: ...**` checkbox syntax.** The previous workflow-orchestrator plan uses this format. It maps cleanly to "executable by an agentic worker, with each step being a discrete checkable action."

## Architectural Context

- **The orchestrator is a markdown instruction file, not a state machine.** `.claude/skills/feature/SKILL.md` is prose that Claude reads and follows step by step. There is no compiled control flow; sequencing is purely "the model reads the steps in order and executes them." This means the new commit+push step at the end of the post-return protocol is a markdown insertion, not a code change.
- **Agents return to the orchestrator by ending their conversation turn.** There is no return-value protocol — the agent finishes its instructions, the orchestrator's turn resumes, the orchestrator reads `context.yaml`, decides what to do next. This is why the commit+push must happen **inside** the agent's instructions (before it returns) — not in the orchestrator's post-return protocol. The orchestrator only commits the *transition* write (advancing `current_step`).
- **Two distinct writes to `context.yaml` per step boundary.** With this feature: (a) the agent writes its own checkpoint / artifacts / lists / etc. and commits → push → returns; (b) the orchestrator advances `current_step` and `completed_steps` and commits → push → invokes the next agent. The branch ends each step with two commits, not one. The acceptance criterion ("at least one commit per pipeline step ... plus orchestrator transition commits") confirms this is intentional and acceptable.
- **The push-failure escalation halts before the next agent is invoked.** Because the orchestrator reads `workflow.escalated` after every agent returns (see `.claude/skills/feature/SKILL.md` line 62), an agent that fails to push and writes the escalation will be caught by the existing halt logic without any orchestrator changes. The orchestrator's *own* push, however, happens **after** the post-return protocol — if that push fails, the orchestrator must halt itself before invoking the next agent. This means the orchestrator's push-failure branch needs to be explicitly added to the orchestrator's instructions, separate from the agent push-failure branches.
- **The draft PR is created on an empty branch today.** The Define agent today runs `git push -u origin <branch>` then `gh pr create --draft`. After this feature, the push step is preceded by a commit step, so the PR's first commit will be Define's spec commit. This is the spec's Constraints item 3 — verified by inspecting `.claude/agents/define.md` lines 30-31. No additional change needed beyond the new commit step itself; the PR creation step works the same way with or without prior commits.
- **The Implement agent already pulls before working.** Lines 31-35 of `.claude/agents/implement.md` describe a `Pre-Implementation Setup` block that runs `git pull` if the branch tracks a remote. This means by the time Implement starts, the remote is the source of truth for the previously-completed steps — Define, Research, Plan commits will already be on the remote because each of those agents pushed before returning. This pre-existing pull step is unchanged; the new behavior just makes its inputs richer (the remote will actually have content).
- **Hook is non-blocking.** `.claude/hooks/skill-check.sh` emits a `[skill-check]` reminder via `hookSpecificOutput.additionalContext` but does not exit non-zero — the Bash call proceeds either way. The hook is a behavior nudge, not a gate. This means the new commit steps will not fail if `Skill(git-commit)` is forgotten — they will just produce a noisier transcript. The instruction text must therefore be explicit ("invoke `Skill(git-commit)` before this commit") rather than relying on the hook to enforce.

## Key Insights for the Planner

- **Six agent files + one skill file + zero new files.** All edits are in-place. There are no new templates, no new directories, no new fixtures, no migration of existing content. The plan's file map will have a `Modified` list of seven entries and an empty `New Files` / `Deleted Files` list.
- **Three of the six agents need their frontmatter `skills:` array extended to include `git-commit`.** Research, Validate, Document. Without this, the agent cannot invoke `Skill(git-commit)` and the hook's `[skill-check]` reminder fires into a dead end. This is a small, easy-to-miss frontmatter change that must be in the plan.
- **The Implement end-of-step commit is conditional.** Acceptance criterion 4 says "only if dirty". The instruction must include a `git status --porcelain context.yaml` check (or equivalent) and only stage-commit if there's a delta. Without this, the agent could produce an empty commit, which violates the spec's Constraints item 4.
- **The orchestrator's commit happens twice per step boundary in spirit.** Read `.claude/skills/feature/SKILL.md` lines 56-64: today the protocol is invoked after every agent return except inside the Approval Gate's revision pass. The new commit+push at the end of the protocol therefore runs after Define, Research, Plan, Implement, Validate, and Document — six transition commits per feature. Plus the Approval Gate's transition from Define→Research (which today says "skip the post-return protocol on the revision pass — only run it on approval") — so the approval gate's "run the post-return protocol" call now includes the new commit step automatically. No special-casing needed for the approval gate; the post-return protocol just gets longer.
- **All push-failure escalations look identical.** A single boilerplate paragraph can be copied into each of the seven instruction files. Phrasing should match the existing escalation YAML blocks in `implement.md` (lines 51-59) and `validate.md` (lines 65-73). The `# Merge into existing workflow block — do not replace other fields` comment should be preserved verbatim.
- **Acceptance criterion 11 (end-to-end verification) is the hardest to satisfy mechanically.** Running the full pipeline on a sample feature is the proof, but doing so consumes real LLM time and requires a real repo state. The plan should propose either (a) a manual dry-run task where the implementer runs `/feature` on a trivial throwaway idea and inspects the resulting `git log` for one commit per step, or (b) a documentation check that asserts each agent's instructions contain the required commit + push steps. Option (b) is more reproducible; option (a) is closer to what the spec asks for. The Planner should pick one and be explicit.
- **The plan's commit messages are themselves the feature.** Because this entire feature is instruction prose that mentions commit messages, the plan's example commit messages will be parsed as documentation by anyone reading the agent files later. They must be exactly the strings the spec lists in its Requirements section (e.g. `docs(spec): add spec for <feature name>`, `chore(context): advance workflow to <next step>`) — not paraphrases.
- **The `git-commit` skill's hook will fire on the implementer's own commits while implementing this feature.** Because the implementer will be editing files inside `.claude/` that touch agents and then running `git commit` to commit those edits, the hook reminders will fire during implementation. This is expected, not a problem — just worth noting so the implementer is not confused.
- **There is no existing test for instruction-file changes.** The repo has no tests at all. Verification of correctness is by file inspection. The plan should call this out so the implementer knows the standard verification (run tests, check coverage) does not apply here and the implement skill's coverage gate is moot for this feature.

## Open Questions

- **Who flips `1_spec.md`'s `Status:` field from `Draft` to `Approved`?** The Research agent's gate at `.claude/agents/research.md` line 23 requires `**Status:** Approved` in `1_spec.md`, but neither the Define agent nor the orchestrator currently has an instruction that edits the spec file to change this. This is a pre-existing gap unrelated to the commit-on-handoff feature, but if the user approves the spec at the gate and the orchestrator doesn't flip the status, the Research agent's gate will reject. The Planner does not need to fix this — but should note that if the approval-gate flow currently works in practice (the previous workflow-orchestrator feature presumably passed it), then the flip is being done implicitly somewhere (possibly by the user manually, or by Claude doing it implicitly during the Define conversation). If the new Define commit step commits `1_spec.md` as `Status: Draft` and the status is flipped later (by the orchestrator on approval), that flip will need its own commit — which the spec does not currently address. **Recommendation:** ask the user during planning whether the orchestrator should also commit the status flip, or whether the status flip is out-of-scope for this feature and stays a manual concern.
- **Should the orchestrator commit message say "advance workflow to <next step>" or "complete <prev step>"?** The spec gives the former (`chore(context): advance workflow to <next step>`). After the Document step, the next value is `complete`, so the commit message would read `chore(context): advance workflow to complete` — which reads slightly oddly. **Recommendation:** keep the spec's exact phrasing; the oddness is minor and consistency wins.
- **Should the Document agent's commit message specify which docs it updated?** The spec gives `docs: update documentation for <feature name>` — a single commit message regardless of how many docs were touched. This works. But the Document agent could in theory produce multiple commits (one per doc surface) if that's preferred. **Recommendation:** keep it as one commit per spec; multiple commits are out-of-scope and would complicate the per-step audit trail.
